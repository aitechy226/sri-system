End-to-End Workflow — Ingestion + Query to Verdict

Phase 1 — Ingestion (one-time, before any query)

1. Documents loaded
  PDFs parsed by layout engine into structured elements: headings, body paragraphs, tables, footnotes.

2. Semantic chunking
  Each element type is chunked differently:
  - Body text: sliding window of 80 words, 15-word overlap between adjacent chunks
  - Tables: atomic — never split, one chunk per table (splitting a table loses row/column context)
  - Footnotes: atomic — one chunk per footnote (FSI documents hide material disclosures in footnotes)
  - Headings: merged with the body text that follows them so section context travels with the content
  Trailing chunks smaller than 20 words are discarded and absorbed into the preceding chunk's overlap.

3. Indexing
  Each chunk is indexed three ways simultaneously:
  - BM25 index built on tokenized chunk text (keyword ranking)
  - TF-IDF matrix built on chunk text (sparse term-frequency scoring)
  - Each chunk passed through the sentence transformer model to produce a dense vector and stored in the vector index (semantic search)

Phase 2 — Query (every time a user asks a question)

4. User submits a query
  e.g. "What is our KYC policy for high-risk customers?"

5. Orchestrator LLM picks the corpus
  Reads the query, selects the right index based on content. Never asks the user.

6. Query vectorization
  The query string is passed through the same sentence transformer model used during ingestion to produce a query vector. Required before dense embedding search can run.

7. Hybrid retrieval runs in parallel
  - BM25 search on raw query text → ranked list of chunks
  - TF-IDF search on raw query text → ranked list of chunks
  - Dense embedding search: cosine similarity between query vector and chunk vectors → ranked list of chunks

8. Reciprocal Rank Fusion merges the results
  Three ranked lists combined into one. Top-scoring chunks returned as the evidence object.

9. Orchestrator evaluates retrieval quality
  top_score > 0.010 → reliable, proceed to synthesis
  top_score 0.003–0.010 → marginal, rephrase using domain synonyms and retry
  top_score < 0.003 → no signal, try different corpus or substantially different query
  Minimum 3 retrieved chunks required before synthesizing — even a high top_score isn't enough if only 1–2 chunks came back. Conservative design: biased toward breadth of evidence before synthesizing.
  Known limitation: for simple factual lookups where the answer genuinely lives in one place (e.g. "what is the SAR filing deadline?"), this can be overly strict — the system retries unnecessarily. In production this threshold should be tunable per corpus type.

10. Orchestrator calls the synthesis tool
  Passes the matched chunks + original query as the evidence object.

11. Synthesis LLM generates the answer
  Input: system prompt (use only retrieved evidence) + matched chunks + original query. That's it — no corpus access, no training knowledge.
  Output: natural language answer. Instructed to use precise language from the source but it's still an LLM — paraphrase drift is possible.

12. _extract_key_claims() scans the answer
  Regex pulls out verifiable sentences — numbers, regulatory terms, obligation language (must/shall/required).
  These extracted claims, not the full answer, are what gets verified.

13. Orchestrator calls the governance tool
  Passes the extracted claims and the original retrieved passages.

14. Governance tool runs word-overlap verification
  Each claim checked independently against raw corpus text.
  grounding_score = verified claims / total claims
  ≥ 0.6 → APPROVED  |  0.30–0.59 → FLAGGED  |  < 0.30 → REJECTED
  Common trigger for FLAGGED: paraphrase drift, not hallucination — e.g. corpus says "one month," synthesis wrote "30 days." Same fact, word-overlap fails.

15. FLAGGED retry
  Orchestrator reads the list of unverified claims from the verdict.
  Constructs a new targeted query on those specific gaps — LLM's own judgment, not a blind re-run.
  Re-searches, re-synthesizes. Maximum 2 attempts — soft limit enforced via system prompt instruction to the LLM, not hard-coded in the router.

16. Final answer delivered
  Verdict + grounding score + audit ID (SHA-256 hash, immutable record of what was retrieved, synthesized, verified).

Key Numbers to Know Cold

GROUNDING_THRESHOLD = 0.6 (default in code)
Retrieval score — reliable: > 0.010
Retrieval score — marginal: 0.003–0.010 (rephrase + retry)
Retrieval score — no signal: < 0.003 (try different corpus)
Min retrieved chunks before synthesis: 3 (breadth of evidence requirement, independent of score)
Governance verdicts: APPROVED / FLAGGED / REJECTED
APPROVED: grounding ≥ 0.6
FLAGGED: grounding 0.30–0.59 — LLM constructs targeted retry query, max 2 attempts (soft limit via prompt)
REJECTED: grounding < 0.30 — insufficient evidence, analyst must add documents and re-run
Chunk size: 80 words | Overlap: 15 words (config.py defaults)
Retrieval: keyword (BM25 + TF-IDF) + semantic (dense embeddings) → Reciprocal Rank Fusion
Reasoning LLM: Claude (orchestrator — never generates content)
Synthesis LLM: separate model, input = matched chunks + query only
